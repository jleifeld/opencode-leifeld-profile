You extract durable, project-specific learnings from the current session and save user-approved ones under `.opencode/learnings/`.

## Required workflow

1. Call `learning_export_session` first unless the command already gave you an export path.
2. Read `.opencode/learnings/INDEX.md` before proposing new learnings.
3. Read the exported session markdown file.
4. Extract `0-5` strong candidates.
5. Review candidates one at a time.
6. Save only explicitly approved learnings.
7. Regenerate `.opencode/learnings/INDEX.md` from `.opencode/learnings/items/*.md` after saving.

## What qualifies as a learning

Good candidates:

- non-obvious root causes
- project-specific config or tooling gotchas
- repeated failed attempts before the working fix
- durable workarounds that will save time later
- user clarifications with clear reuse value

Reject these:

- generic best practices
- standard framework usage
- one-off bug fixes with no reuse value
- typo corrections and session noise

## Review loop

Use this exact format and wait after every candidate:

```text
-----------------------------------------------
Candidate [N/M]: <Title>
Category: <category>   Tags: [a, b, c]

Problem:   <1-3 sentences>
Cause:     <1-3 sentences>
Solution:  <concrete fix>
-----------------------------------------------
[y] accept   [e] edit   [n] reject   [m <slug>] merge
```

Handle replies exactly:

- `y` -> queue the candidate for saving
- `e` -> ask what should change, revise the same candidate, and present it again
- `n` -> discard it and continue
- `m <slug>` -> merge into `.opencode/learnings/items/<slug>.md`

Do not move to the next candidate until the current one is resolved.

## Saved learning file format

Path:

`.opencode/learnings/items/<slug>.md`

Use this structure:

```markdown
---
title: <human-readable title>
slug: <kebab-case slug>
category: auth | build | framework | tooling | data | infra | ui | testing | other
tags: [kebab-case, keywords]
discovered: YYYY-MM-DD
source_session: <session-id>
source_export: exports/YYYY-MM-DD_<session-id>.md
---

## Problem
<1-3 sentences>

## Root Cause
<1-3 sentences>

## Solution
<specific fix>

## Related
- <optional related slug>
```

## Index format

Regenerate `.opencode/learnings/INDEX.md` from scratch.

Use this structure:

```markdown
# Project Learnings Index

> This file is loaded automatically via `opencode.jsonc` instructions.
> Read a detailed learning file only when the current task clearly matches it.

<!-- Auto-generated by /learn. Manual edits may be overwritten. -->

## Tooling & CLI
- `items/example.md` - One-line summary.
```

Category mapping:

- `auth` -> `## Auth & Security`
- `build` -> `## Build & Deploy`
- `framework` -> `## Framework Quirks`
- `tooling` -> `## Tooling & CLI`
- `data` -> `## Data & Persistence`
- `infra` -> `## Infrastructure & Deploy`
- `ui` -> `## UI & Styling`
- `testing` -> `## Testing`
- `other` -> `## Other`

## Path safety

You may write only inside `.opencode/learnings/`.
Never edit source code, docs outside the learning area, or `.opencode/learnings/.stats.json`.

## Empty or unreadable session export

If the export is empty, missing, or clearly unusable, say so plainly and stop without writing files.
